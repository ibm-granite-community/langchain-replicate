# Project Architecture Rules (Non-Obvious Only)

- The package is schema-driven at runtime: [`ReplicateBase._input_properties`](../../src/langchain_replicate/_base.py:140) sorts model OpenAPI input fields by `x-order`, and both [`Replicate._create_prediction_input()`](../../src/langchain_replicate/llms.py:101) and [`ReplicateEmbeddings._create_prediction_input()`](../../src/langchain_replicate/embeddings.py:58) derive their primary input key from that schema instead of hardcoding one.
- Deployment support is not just routing: [`ReplicateBase._deployment`](../../src/langchain_replicate/_base.py:111) resolves the current release and mutates [`ReplicateBase.model`](../../src/langchain_replicate/_base.py:78) to the released model name, so downstream logic must tolerate model identity changing after initialization.
- Tool-calling reliability depends on a bidirectional compatibility layer: outbound payloads go through [`_adjust_prediction_input()`](../../src/langchain_replicate/_base.py:52), while inbound assistant tool arguments are repaired by [`_normalize_tool_arguments()`](../../src/langchain_replicate/chat_models.py:79) before LangChain parsing.
- Stop handling has two architectures: native schema-backed stop fields via [`ReplicateBase._stop_input()`](../../src/langchain_replicate/_base.py:168), and local truncation/cancellation in [`Replicate._stream()`](../../src/langchain_replicate/llms.py:145) when the backend exposes neither `stop` nor `stop_sequences`.
- Embeddings are intentionally model-shape agnostic: [`ReplicateEmbeddings.texts_value_mapping`](../../src/langchain_replicate/embeddings.py:37) exists because some Replicate embedding models expect serialized or joined text rather than `list[str]`.
- Type checking is pyrefly strict mode (not mypy/pyright); the `# pyrefly: ignore [...]` suppression pattern is project-standard for Replicate SDK pydantic-v1 interop.
